# Design — Output hygiene for generated book content

Date: 2026-07-08
Status: Approved-pending-review
Scope: Item 1 of the owner's UI/content session. (Item 2 — the owner's custom
romance workflow — is a separate brainstorm; this spec does not cover it.)

## Problem

Chapter and canon files generated by the default "Novel" pipeline contain
chatbot framing that belongs only in a chat window, never in a pure
API-generated artifact. Two observed leaks (Chapter 1, "My Sixth Test Medical
Romance"):

- **Example 1** — a bare step-label header leaked as the first line:
  `Polish Chapter 1` above `Chapter 1: The Weight of Steel`.
- **Example 2** — a trailing chat epilogue: a `Next Steps:` heading, a bulleted
  list of "directions" (Expand on the Hemorrhage / Develop Elias's Internal
  Conflict / World Bible Expansion / Character-to-Character Dynamic), and the
  solicitation `Which direction would you like to prioritize moving forward?
  I'm eager to continue building this scene!`.

## Root cause

Two layers are meant to keep generated files clean; both have gaps.

**Layer 1 — the prompts.** A full audit of every step prompt (all ~30
`library/pipelines/*.json` plus the hardcoded builders in
`gateway/src/services/projects.ts`) shows the modern pipelines (MSF, nerdynovelist
stages 2–5, romance-spicy/sweet, romantasy-production, technothriller-production,
humanize, all editorial) consistently end each step with an explicit output
contract ("Output only X — no commentary/preamble/changelog"). The **older
default pipelines predate that convention** and are the only leak sources:

- Group A (default Novel prose): `writeChapterPrompt` in `projects.ts` (feeds both
  `buildNovelPipeline` and the `book-production.json` Write step) and
  `book-production.json`'s Write Chapter step — PARTIAL (say "write prose" but do
  not forbid commentary). In `buildNovelPipeline` there is no Polish step after
  the draft, so the leaky draft is the retained output. This is Example 2.
- Group B (canon builders consumed verbatim downstream): the 5 steps of
  `book-bible.json`, the 6 steps of `book-planning.json`, and the 13
  planning/bible/outline steps built in `buildNovelPipeline`.
- Group C (other legacy): `book-launch.json`, `format-export.json`, the 21
  analysis steps in `deep-revision.json`, and the PARTIAL council/ideation steps
  (`editorial-outline-council`, `msf-phase1`, `msf-phase2` review steps,
  `romantasy-planning` evaluate/select, `technothriller-planning` grounding,
  `nerdynovelistai-stage1` dossier steps).
- Excluded by design: every "Compile manuscript / completion report" step —
  commentary (strengths, next steps) IS the deliverable there; hardening would be
  wrong.

**Layer 2 — the post-processor** (`gateway/src/services/strip-meta.ts`,
`stripMetaCommentary`). Deliberately conservative because it runs on every saved
step. It catches `# Polish Chapter N` (markdown-heading form only), leading
acknowledgement interjections, and a fixed set of trailing phrases ("would you
like to proceed", "shall we continue", "let's keep momentum"). It does **not**
catch the bare `Polish Chapter 1` label (no `#`) or the Example 2 epilogue.

## Solution — two layers plus a self-enforcing guard

### Layer 1 — prompt hardening (root cause, primary fix)

Append an output-contract clause to every flagged **non-report** step, matched to
its output type, reusing the exact phrasing the protected pipelines already use:

- Prose steps: "Output the complete chapter prose only, in Markdown. No
  commentary, no preamble, no 'next steps', no questions to the reader."
- Bible / outline / analysis steps: "Output only the [world bible / outline /
  critique]. No preamble, no commentary, no 'would you like…' epilogue."

Mechanical edits — add a clause, change no control flow. Files touched:
`gateway/src/services/projects.ts` (`writeChapterPrompt` and the inline prompt
strings in `buildNovelPipeline`) and the flagged `library/pipelines/*.json` steps
in Groups A–C. Report/compile steps are left unchanged.

### Layer 2 — step-aware strip-meta (safety net)

Extend `stripMetaCommentary` with an optional `prose` flag threaded from the known
call sites (`index.ts`, `projects.routes.ts`) so prose (chapter) outputs get an
aggressive strip while reports/outlines keep today's conservative behavior:

- Leading (all callers): also strip a **bare** `Polish Chapter N` / `Write
  Chapter N` label (no `#`). Fixes Example 1.
- Trailing (all callers): add only unambiguous conversational solicitations —
  "which direction would you like…", "I'm eager to continue…", "let me know which
  direction", "happy to keep going". These never occur in a legit report or in
  prose.
- Trailing (`prose: true` only): also pop a trailing block whose first line is a
  bare `Next Steps:` / `Chapter Ending Hook:` heading. Gated behind `prose`
  because `Next Steps:` is legitimate in the completion report and `Chapter
  Ending Hook:` is legitimate in the chapter outline (the outline prompt asks for
  one per chapter). This fully cleans the Example 2 middle block as a net even if
  a prompt leak slips through.

The call sites that process chapter/prose step output pass `prose: true`; the
chat path and report/outline paths keep the default (conservative) behavior.

### Guard — self-enforcing invariant

Add a unit test (following `library-pipeline-skill-refs.test.ts`) that walks every
`library/pipelines/*.json` step and asserts every non-report step's
`promptTemplate` contains an output-contract phrase (a small regex/keyword set:
`output only|output the complete|no commentary|no preamble|prose only|...`).
Report/compile steps are allow-listed out. This makes the "same protection
everywhere" request self-enforcing: any future step — including the owner's new
romance workflow — fails the build if it lacks an output contract.

## Testing (verification per step)

Existing harnesses are extended (TDD — write the failing assertion first):

1. `tests/unit/strip-meta-commentary.test.ts` → add cases: bare `Polish Chapter
   1` leading label stripped; Example 2 epilogue stripped with `prose: true`;
   report text with a legit "Next Steps:" section preserved with `prose: false`.
   Verify: `npm run test:unit` (these cases).
2. `tests/unit/production-prompts.test.ts` → assert `writeChapterPrompt` output
   contains the no-commentary contract. Verify: `npm run test:unit`.
3. New guard test → every non-report pipeline step has an output contract.
   Verify: it fails before the JSON edits, passes after.
4. Full suite green: `npm run test:unit` (currently ~1296 tests) + `npx tsc
   --noEmit`.
5. End-to-end (Mercury): regenerate one chapter via the default Novel path and
   confirm the saved file contains neither the bare label nor the epilogue.

## Out of scope

- Item 2, the owner's custom romance workflow (separate brainstorm/spec).
- The deferred B7 TODO (mid-document fenced AI reasoning) — related but a
  different mechanism (chapter-level validate/repair); not addressed here.
- Report/compile step wording.

## Rollout

Code + content edits are local-tree; no migration. Deploy is the normal
`touch build_now` (Mercury) then manual backup-first compose (Neptune). Existing
already-generated files are not retroactively cleaned (regeneration only).
