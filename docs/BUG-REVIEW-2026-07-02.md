### 1. [HIGH] Bearer-auth gate is case-sensitive but Express routing is case-insensitive → auth bypass via uppercased path
**`gateway/src/index.ts:458`**  ·  verdict: **confirmed**  ·  corroborated by 2 finders (api-core, core-entry)

The API auth middleware (and the IP/rate-limit gates) decide whether to enforce auth with `req.path.startsWith('/api/')` (lines 418, 458), a case-sensitive string check. Express 4 (confirmed express@4.22.2 in node_modules) matches routes case-INSENSITIVELY by default (no `case sensitive routing` is set anywhere). All API routes are registered lowercase (e.g. `app.post('/api/projects/:id/...')`). Therefore a request to `/API/projects/...` (or `/Api/...`) does NOT start with lowercase `/api/`, so the auth middleware calls `next()` and skips the bearer check entirely — yet the same request still matches the lowercase route handler and executes it. I reproduced this empirically against a minimal express@4.22.2 app in this repo's node_modules: `/api/status` → 401 (blocked), `/API/status` and `/Api/status` → 200 with the handler body returned. Because the IP allowlist defaults to allow-all, any LAN host can reach every mutating and secret-exposing `/api/*` endpoint unauthenticated simply by changing the case of the `/api` segment. The rate-limit gate (line 418) is defeated the same way. This is the exact 'auth bypass' the LAN threat model treats as in-scope.

**Failure scenario:** With auth enabled, an unauthenticated LAN client sends `GET http://host:3847/API/status` or `POST http://host:3847/API/projects` — the auth middleware's `startsWith('/api/')` is false so it passes the request through, but Express case-insensitively matches the lowercase route and runs it, returning data / performing the action with no token.

**Suggested fix:** Make the gate case-insensitive to match Express's routing, e.g. `if (!/^\/api\//i.test(req.path)) return next();` in all three middlewares (auth, IP, rate-limit), or set `app.set('case sensitive routing', true)` so routing and the gate agree. Prefer the former so existing lowercase routes keep working.

**Verifier:** Traced and empirically reproduced end-to-end against this repo's express@4.22.2. Routes are registered lowercase directly on the app (e.g. app.get('/api/status'), app.post('/api/chat') in gateway/src/api/routes/core.routes.ts). No 'case sensitive routing' is set anywhere in gateway/src (only app.set('trust proxy') at index.ts:376), so Express's default case-INSENSITIVE matching applies. The rate-limit gate (index.ts:418), bearer-auth gate (index.ts:458), and 404 handler (phase-11-http.ts:70) all decide enforcement with case-sensitive req.path.startsWith('/api/'). Express does not lowercase req.path, so a request to /API/status has req.path==='/API/status', startsWith('/api/') is false, the auth middleware calls next() and skips the bearer check — yet Express still matches the lowercase route handler and runs it. Repro: GET /api/status -> 401 blocked, GET /API/status and /Api/status -> 200 with handler body; POST /API/projects/x/advance -> 200 {mutated:true} while /api/... -> 401. With the IP allowlist defaulting to allow-all (index.ts:406, message at 390), any LAN host reaches every mutating and secret-exposing /api/* endpoint unauthenticated by upper-casing the /api segment; rate limiting is defeated identically. Could not refute. Severity corrected from critical to high per the repo's single-user trusted-LAN threat model (adversary is other LAN hosts), but it is a complete, trivial bypass of the designed bearer-auth perimeter.

### 2. [HIGH] Review-resolver sweep's driveProject bypasses the per-project auto-execute lock, double-driving the same step against the dashboard runner
**`gateway/src/init/phase-10-heartbeat-bridges.ts:263`**  ·  verdict: **confirmed**  ·  area: init-security

The 60s sweep (lines 263-275) resumes approved human-review gates and immediately drives the project via `driveProject` -> `commandHandlers.startAndRunProject`, guarded only by `!heartbeatDriving`. It never consults the `autoExecuteInFlight` per-project lock in gateway/src/api/routes/projects.routes.ts:14-17, which exists precisely because "two runners processing the same active step -> duplicated chapters + double cost". The studio PipelineRail polls every 3-15s and kicks POST /api/projects/:id/auto-execute whenever it sees an active phase (frontend/studio/src/components/write/PipelineRail.tsx:80). After a gate approval, the sweep and the browser runner both find the SAME 'active' step (project.steps.find(s => s.status === 'active')) and both generate it. When both call completeStep for that step, the second call re-completes it, overwrites step.result with whichever output finished second, and its runnableSteps() scan activates step N+2 while N+1 is still in flight (an 'active' non-parallel step does not block the pending frontier, projects.ts:1041-1123) — so N+2's prompt is enriched via enrichWithPriorResults WITHOUT N+1's result. The headless hook (line 289) at least documents "Assumes headless use (no concurrent browser driver)", but the sweep runs unconditionally on every deployment and its stated purpose is dashboard-only deployments — i.e. exactly the case where a browser driver is present.

**Failure scenario:** User has the Write view open (PipelineRail polling) and approves a pending human-review gate in the dashboard. Within 60s the sweep resumes the project and starts driving it server-side; within 3s the browser poll also fires /auto-execute (lock is free — the sweep never took it). Both drivers generate the same chapter step concurrently: double AI spend, the step's saved result overwritten by the slower response, and the double completeStep prematurely activates the step after next, whose prompt is built without its predecessor's content — out-of-order/corrupted chapter continuity for the rest of the run.

**Suggested fix:** Share one per-project in-flight lock between the /auto-execute route and driveProject (e.g. move autoExecuteInFlight to ProjectEngine or a shared module; driveProject skips or waits when the project is already being driven, and /auto-execute returns 409 while driveProject holds it).

**Verifier:** Confirmed end-to-end. autoExecuteInFlight is declared/used only in projects.routes.ts (lines 17,616,619,624,1047) and taken solely by POST /api/projects/:id/auto-execute. The 60s sweep (phase-10-heartbeat-bridges.ts:255-262) drives via commandHandlers.startAndRunProject (index.ts:2173), which never touches that Set — so the per-project lock does NOT serialize the sweep against the browser runner. resolveReviewGates→applyReviewResume (projects.ts:899) synchronously flips the next step to active and the sweep immediately drives it, so the step is 'active' and being generated server-side for the whole (multi-second/minute) AI call. The reviewer overstated one leg: the frontier poll at PipelineRail.tsx:80 is guarded by kickedRef (once per project id per mount), so a tab open through the gate won't re-kick. But the bug still reaches via (a) a fresh mount / opened-to-approve (kickedRef null → line 80 kicks after the sweep flips status to active — the normal human-review workflow) and (b) the unguarded manual continue button at PipelineRail.tsx:382. Result: same chapter step generated twice concurrently → double AI spend, step.result overwritten by the slower response, and downstream steps enriched against a result that changes underneath them. Manuscript corruption + double cost. Severity high is correct (needs a human-review gate + a specific interleaving, so not critical, but it is manuscript data corruption which the repo grades HIGH+).

### 4. [HIGH] Word-target continuation passes never show the model its own draft — 'continue from where you left off' is sent with zero prior prose
**`gateway/src/index.ts:2356`**  ·  verdict: **confirmed**  ·  area: ai-bridges

Both continuation call sites (index.ts:2354-2367 bridge path, projects.routes.ts:751-767 studio path) send the prompt 'Continue writing from where you left off... Do NOT repeat what was already written' via gateway.handleMessage on the 'goal-engine'/'project-engine' channels. Those channels are in the isProjectChannel list (index.ts:791), so handleMessage builds a CLEAN messages array containing only the continuation instruction (index.ts:813-814) — no conversation history. The reused `projectContext` was built once before the initial generation and (per buildProjectContext, projects.ts:1420) contains only COMPLETED steps' results, never the current step's partial aiResponse. The model therefore has no way to know what it already wrote; it can only re-start the chapter from the outline. appendContinuation's overlap dedup (wordcount.ts:21) needs a >=40-char exact tail/prefix match, which a blind restart almost never produces, so the restarted text is appended verbatim with '\n\n'.

**Failure scenario:** A novel-pipeline chapter step with wordCountTarget 4000 gets a 2500-word first pass. runWordTargetContinuation fires a continuation; the model, seeing only the outline context and the instruction, re-writes the chapter's events from the top. The result is appended, producing a chapter file whose second half re-narrates the first half with different wording — silently saved to the book's data/ dir and assembled into the manuscript.

**Suggested fix:** Include the tail of the accumulated text (e.g. last 2-4K chars) in the continuation prompt, or pass the draft-so-far as an assistant message in the messages array for continuation calls.

**Verifier:** Traced end-to-end and could not refute it. Both continuation call sites (index.ts:2355 bridge, projects.routes.ts:757 studio) reuse a projectContext string built once at index.ts:2204 — before generation — via buildProjectContext, whose writing-phase branch (projects.ts:1642/1698) filters strictly to steps with status==='completed' && result. The current chapter step is still active and its output lives only in the local aiResponse variable (never written to step.result until the block finishes), so it is categorically absent from context. Both calls use the goal-engine/project-engine channel, which index.ts:791 flags isProjectChannel, so index.ts:813-814 builds messages=[{role:'user',content}] — only the 'continue from where you left off' instruction, no history, no assistant turn. projectContext is merely appended to the system prompt (index.ts:785). The model therefore never sees its own partial draft. appendContinuation (wordcount.ts:26-31) requires a >=40-char exact tail/prefix overlap to dedup; a blind restart won't match, so it appends verbatim with '\n\n'. The claimed failure is a genuine consequence: the continuation cannot be seamless because the model has no anchor to what it already wrote; the result (duplicated/discontinuous prose) is silently saved to the book's data/ dir and assembled. Not guarded elsewhere, not dead code, not intentional per any comment. Severity high is correct: it silently corrupts the primary generated-manuscript output on the default novel-pipeline path with no error surfaced.

### 28. [LOW] File-download streams have no error handler — a directory or vanished file leaves the response hanging via an uncaught stream error
**`gateway/src/api/routes/documents.routes.ts:536`**  ·  verdict: **confirmed**  ·  area: api-knowledge-docs-media

### 31. [LOW] iCal all-day events emit DTEND == DTSTART (zero duration), breaking import
**`gateway/src/services/release-calendar.ts:157`**  ·  verdict: **plausible**  ·  area: svc-media-marketing

exportICS() writes each all-day VEVENT with DTSTART;VALUE=DATE:<day> and DTEND;VALUE=DATE:<same day>. Per RFC 5545, for VALUE=DATE the DTEND is exclusive and MUST be after DTSTART; a zero-length all-day event is invalid and Google Calendar in particular refuses to display/import it (Apple Calendar and Outlook are inconsistent too). Since the entire purpose of the .ics export is 'anti-miss safeguards', the reminders the author imports can silently fail to appear.

**Failure scenario:** Author exports the release calendar and imports the .ics into Google Calendar. Every price-pulse/launch event has DTSTART=DTEND=20260701 and is dropped or shown as an empty all-day sliver, so the critical launch-day reminder never surfaces.

**Suggested fix:** Set DTEND to the day after DTSTART (DTSTART + 86400000) when formatting the exclusive all-day end date.

**Verifier:** Code-level defect is confirmed and reachable: exportICS() (release-calendar.ts:156-157) emits DTSTART;VALUE=DATE and DTEND;VALUE=DATE with the identical day (both from dateOnly), producing zero-duration all-day VEVENTs. Per RFC 5545 3.6.1 a VALUE=DATE DTEND is non-inclusive and MUST be strictly later than DTSTART, so this is a genuine spec violation with no guard/normalization anywhere; the route GET /api/calendar/export.ics (knowledge.routes.ts:965) is live behind auth, not dead code. However the reviewer's concrete impact ('Google Calendar refuses to display/import; the launch reminder never appears') is overstated — modern Google Calendar is lenient and typically imports DTEND==DTSTART all-day events on the start day; behavior across clients varies. So the malformation is confirmed but the reminder-loss outcome is plausible, not demonstrated. Severity: this is a non-destructive convenience export (events still live in BookClaw, no manuscript data at risk) with a trivial one-line fix, so low is more appropriate than medium.
