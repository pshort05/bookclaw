## North Star — the ultimate goal (use this to weigh every other decision)

**This is the destination the project is steering toward. It is not a near-term task — it is the lens for evaluating other work.** When scoping any non-trivial change (refactors, the rename, new features, storage shapes, API design), prefer the option that moves toward — or at least does not move away from — this end state. Where a decision is a toss-up on its own merits, the tie-breaker is "which choice makes the North Star cheaper to reach later."

**The end state:** BookClaw runs as a **multi-author, multi-book studio**. Several books are in flight at once, each owned by a distinct **author identity**, written in its own **genre**, produced by its own **customizable pipeline** — e.g. a Romantasy from author A, a contemporary romance from author B, and a mundane sci-fi from author C, all simultaneously. Adding another author, genre, or pipeline variant should be **configuration, not code changes**.

**The four entities this implies (none exist as first-class concepts today):**

- **Book** — owns state across `planning → bible → production → revision → format → launch`, references an author profile + genre profile + pipeline, and carries its own history. Today `workspace/projects/<id>/` is a flat list with no "book" that spans phases.
- **Author profile** — a named identity bundle (its own SOUL/STYLE-GUIDE/VOICE-PROFILE/PERSONALITY), selectable per book, cloneable/editable without touching the global `workspace/soul/`. Today there is exactly one author identity, hardcoded to `workspace/soul/`.
- **Genre profile** — a named pack (tropes, beats, reader-expectation notes, comp-title pointers), selectable per book, replacing the free-text `genre` field. Today `genre` is free text on each project.
- **Customizable pipeline** — editable, data-driven, versioned step definitions: add/remove/reorder steps, edit each step's prompt, change its `taskType`/tier and `wordCountTarget`, attach skills — **all without editing code**. Selectable per book (and differing per author/genre). Today the 6 templates + `novel-pipeline` are hardcoded in `services/projects.ts`.

## Full security review (post-LAN-exposure)

Opening the bind from loopback to LAN turned several previously-defense-in-depth
assumptions into single points of failure. A proper security review should
cover at minimum:

- [ ] **Bearer token is readable from the public dashboard HTML.** The auth gate covers only `/api/*`; the dashboard at `/` and its static assets are public (`gateway/src/index.ts:280`), and `/` serves its HTML with the token injected (`__BOOKCLAW_AUTH_TOKEN__` placeholder). So in the **default posture** (auth token set, `BOOKCLAW_ALLOWED_IPS` unset) anyone who can reach the port can `curl http://host:3847/`, read the token out of the HTML, then authenticate — the bearer token does **not** stop a deliberate LAN actor; only the source-IP allowlist or the host firewall does.
  - **Threat model (decided):** single-user home LAN, occasional family curiosity. A casual family member won't view-source to extract the token, so token-injection is adequate *against that actor*. This is also why bearer-token was chosen over HMAC/mTLS (rejected as disproportionate). The exposure is therefore a **known, accepted limitation** for the home-LAN deployment — recorded here so it's a conscious decision, not a silent gap.
  - **What actually closes it** for a stronger threat model (e.g. VPS / untrusted LAN): set `BOOKCLAW_ALLOWED_IPS` to the trusted source IP(s)/CIDR(s) — this gates `/` too, in front of auth — and/or enforce at the host firewall / provider security group (the most robust control; mind the Docker bridge caveat that masks source IPs). A reverse proxy doing its own auth + TLS in front is the belt-and-suspenders option.
  - **If it ever needs a code fix:** gate `/` behind auth and hand the dashboard its token a different way (e.g. a one-time login form that sets an httpOnly cookie, or a short-lived session token), so the long-lived bearer token is never embedded in publicly-fetchable HTML. Out of scope for the current threat model — only revisit if the deployment posture changes.
  - **Related minor notes:** the `?token=` query fallback (native `<img>`/`<a>`/`<Audio>` GETs) puts the token in URLs, which proxies / access logs / browser history can capture — low risk on a trusted LAN but worth eliminating if `/` is ever hardened. And there is no TLS (plain HTTP on the LAN); a reverse proxy is the documented path when that matters.
- [ ] **Audit logging: source IP + native Linux log manager.** *(Deferred 2026-05-30 — design investigated, implementation parked.)* Two related pieces:
  - **(original) Audit-log the request source IP** on every gateway HTTP route and every WebSocket connection, so a compromise is forensically traceable. `workspace/.audit/` currently logs `websocket_connected` with socket id only. The IP plumbing already exists (`socketClientIp()`, `req.ip` with `trust proxy` set — from the source-IP allowlist work).
  - **(new ask) Route audit events to the native Linux log manager, with an off switch.** So events land in the OS-managed log (journald / syslog) with its rotation/retention/forwarding, not just the in-workspace file.
  - **Key finding (drives the design):** BookClaw is **Docker-first**, and the `node:22-slim` image has **no in-container journald or syslog** (`/dev/log` absent). So the container-native way to "rely on the native log manager" is to **emit structured audit events to stdout/stderr** and let the platform capture them: under a systemd unit, stdout → journald automatically; under Docker, stdout → the logging driver, which the user points at the host by setting `logging: driver: journald` (or `syslog`) in `docker/docker-compose.yml` (currently the default `json-file` driver, no custom logging). Writing directly to a `/dev/log` syslog socket would NOT work in the default Docker setup (needs `/dev/log` bind-mounted or a remote UDP syslog host), so it's the wrong default for this app — at most an advanced opt-in.
  - **Existing state to reconcile:** there is already a **hash-chained JSONL** audit log (`gateway/src/security/audit.ts` → `workspace/.audit/<date>.jsonl`, tamper-evident), the single sink behind `this.audit.log(category, action, data)` used app-wide (ip_blocked, confirmation events, etc.). The native target would be an **added sink**, not a replacement — journald/syslog don't hash-chain, so keeping the JSONL preserves tamper-evidence. (Lean: keep both, each independently toggleable.)
  - **Open decisions (all deferred):** mechanism (structured stdout *[recommended, cross-env, no dep]* vs. direct syslog socket vs. both); JSONL fate (keep both *[recommended]* vs. native-replaces vs. native-primary); off-switch scope (native sink only *[recommended, e.g. `BOOKCLAW_AUDIT_SYSLOG=0`]* vs. all audit logging vs. per-sink toggles). Implementation lands as a second sink inside `AuditLog.log()` plus the source-IP enrichment; add a smoke-test assertion (e.g. an audit line appears on stdout and carries the client IP).
- [ ] **Vault key on a multi-host Docker volume.** Verify `config/.vault/vault.enc` permissions survive volume mounts. Document the backup-and-restore path for `BOOKCLAW_VAULT_KEY` since losing it means losing every stored credential.
- [ ] **Outbound egress posture.** The AI router and `ResearchGate` make outbound calls to Anthropic / Gemini / OpenAI / DeepSeek / OpenRouter / allowlisted research domains. Confirm `helmet`-level CSP doesn't accidentally weaken egress controls. (Note: `connectSrc` was tightened to `'self'` on 2026-05-30 — but CSP governs the *browser* dashboard, not the server's own outbound AI/research calls, so this item is about the server-side egress path, not CSP.)
